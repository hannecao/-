# 单章小说生产工作流

## 1. 先定义本章

处理章节前，先写一句话：

> 本章通过______，让______从______变成______，并在结尾留下______。

然后只选一个主类型：

- 人物对手戏型：重点是双方目标、潜台词、关系转折。
- 事件链解决型：重点是一个具体问题的因果链、升级和解决。
- 感情推进型：重点是关系只推进一步，不跨级。
- 真相揭露型：严格区分主角可知、读者可知和必须隐藏的信息。
- 过渡整顿型：收束上一阶段，引出下一阶段，不让新冲突抢戏。

若一句话功能写不清，先不要调用 GLM。

## 2. 最小读取

必须读取：

1. 小说生成规范
2. `CLAUDE.md` 或进度文件
3. 当前章细纲
4. 上一章结尾 1500—3000 字
5. 本章核心人物卡
6. 本章直接相关的专名和世界规则
7. 最接近本章类型的优秀 prompt 范例

只有出现疑点时，才继续读取专项设定。禁止默认读取全部章节、全部人物卡和全部世界观。

## 3. 建立章节设计卡

创建 `chXXX_design.md`，固定写：

- 章节类型
- 一句话功能
- 唯一戏剧发动机
- 主角目标
- 核心阻力
- 转折
- 爽点或情绪落点
- 章末钩子
- 本章禁止事项

事件型章节必须先写完整因果链：

> 已发生的改变 → 具体环节错位 → 现场问题 → 问题升级 → 主角查因 → 主角处理 → 暂时结果 → 钩子

不得并列两条乱子。

人物对手戏必须明确：

- A 想得到什么
- B 想得到什么
- 表面谈什么
- 实际争什么
- 哪个可观察瞬间改变双方关系

## 4. 创建 GLM 提示词

创建 `chXXX_prompt.txt`，顺序固定：

1. 写手身份、题材、章节功能、情绪尺度
2. 铁律格式
3. 本书最小必要背景
4. 本章相关暗线逻辑与禁词
5. 前情衔接
6. 本章人物“表演规则”
7. 抽象机制的可观察表现
8. 目标、冲突、转折、钩子、伏笔
9. 场景、连贯、篇幅和输出约束

### 铁律格式至少包含

- 人称与唯一视角
- 特殊角色台词格式
- 禁止章节号和现实元信息
- 年龄数字限制
- 现代梗密度
- 时间、地点、走位
- 每场至少一个非视觉感官细节
- 目标篇幅
- 只输出正文

### 人物信息只写本章需要的

- 外貌或感官锚点
- 说话方式
- 当前目标与情绪
- 已知/未知信息
- 与其他人的当前关系
- 本章允许的变化
- 绝不能写成什么样

### 抽象要求必须落地

不要只写“表现光环失灵、蝴蝶效应、世界偏移”。必须改写成正文能直接呈现的现象或事件。通常给 2—4 种表现，要求 GLM 只选一两个，禁止全堆。

## 5. Prompt 预检

调用 GLM 前检查：

- 是否只有一条主线
- 单视角与结尾要求是否冲突
- 篇幅是否容得下事件数量
- 主角是否亲自完成核心判断和解决
- 是否让男主、系统或高阶人物替主角解题
- 是否新增未经定案的机制、异象、专名或背景
- 是否让人物提前知道真相
- 钩子是否由本章自然触发
- 是否把下一章内容提前塞入

预检失败，先改 prompt。

## 6. 调用 GLM-5.2

沿用项目已经验证可用的 `glm_call.py` 或等效脚本：

- 输入：`chXXX_prompt.txt`
- 输出：`chXXX_raw.md`
- 模型：`glm-5.2`

raw 文件不得直接覆盖正式章节。

若初稿核心结构错误，先找 prompt 的缺口，修正后最多重调一次。Codex不得自行重写整章。

## 7. 复核 raw 初稿

按顺序检查：

### 硬格式

视角、人称、系统台词格式、禁词、元信息、输出格式、篇幅。

### 五要素

目标是否出现；冲突是否成立；转折是否由事件产生；钩子是否自然；伏笔是否只种到规定深度。

### 单一主线

删除第二乱子、无必要大场面、抢主线的感情戏、配角支线和异象堆叠。

### 主角高光归属

核心问题必须主要由主角判断、选择、调查、承担和解决。配角最多提供有限信息，不能给完整答案。

### 暗线泄露等级

严格区分现象、疑点、猜测、证据、答案。系统不得一次讲清世界规则。

### 文风

- 吐槽来自人物处境，不是强行抖包袱
- 同一梗不连续复用
- 不堆模型式金句
- 不反复解释读者已看懂的主题
- 不为了轻喜而破坏人物或场景真实性

## 8. 修订决策

Codex可以修：短句、短段、专名、视角、连续性、信息差、重复说明、少量压缩、删除小支线和小异象。

必须退回 GLM：核心事件链错误、主角高光被抢、两条主线无法靠删减分离、关系方向错误、伏笔答案提前说破、整体文风错误、超过 10% 需要重写。

审稿原则：删减优先，不得提出“可以再增加……”型建议。

## 9. 篇幅是硬约束

- 超出不超过 10%：局部压缩
- 超出 10%—25%：压缩重复、解释和无功能场景
- 超出 25%：不得原样定稿，向 GLM 提交压缩任务

不得以“保留气口”为理由取消字数要求。

## 10. 正式落盘与同步

正式章节不得包含 prompt、审稿说明、模型自评、修改批注或现实元信息。批注另存 review 文件。

章节完成后增量更新：

- `CLAUDE.md` 或进度文件
- 人物当前状态
- 专名表
- 未回收伏笔
- 当前剧情进度
- 下一章必须继承的限制

本章发现的 GLM 常见错误，要转化为下一章 prompt 的明确禁止事项。
